# 线索来源整理与接入计划

## 当前目标

- 先整理哪些平台适合用于反推制造业数字化线索
- 不改 OpenClaw 的现有规则与提示词
- 后续按平台逐个验证，确认能否像 BOSS 一样接入服务器查询任务
- 最后把各平台结果与 OpenClaw 日报结果汇总到同一数据库中查看

## 正确的实施顺序

### 第一步：来源整理

- 把来源分成四类：
  - 招聘
  - 社交内容
  - 企业信息
  - 招采 / 政策
- 每个平台先评估三件事：
  - 是否能在浏览器里稳定访问
  - 是否需要长期登录态
  - 是否适合服务器定时执行

### 第二步：单平台接入验证

- 不直接把所有平台一起接
- 采用和 BOSS 一样的方式，一个一个验证：
  - 能否在服务器 Chrome 中打开
  - 能否在带登录态的情况下稳定查询
  - 能否稳定提取结构化字段
  - 会话失效后是否容易恢复

### 第三步：统一入库

- 各平台不要直接产出各自独立 JSON 结构
- 统一写成标准化信号记录，再落同一个数据库
- 建议统一字段：
  - `platform`
  - `signalType`
  - `companyName`
  - `city`
  - `title`
  - `summary`
  - `evidenceUrl`
  - `publishedAt`
  - `capturedAt`
  - `confidence`
  - `rawPayload`

### 第四步：和 OpenClaw 日报并行查看

- OpenClaw 继续负责日报和研究型输出
- 浏览器任务负责结构化来源抓取
- 两边不要混跑，但可以在数据库层统一查看
- 最终页面按数据类型区分：
  - 日报视图
  - 招聘 / 社交 / 招采 / 企业信息视图
  - 企业聚合视图

## 建议的接入优先级

### 第一批

- BOSS直聘
- 智联招聘
- 中国政府采购网
- 爱企查 / 企查查

原因：

- 信号最直接
- 结构化程度最高
- 更容易验证服务器浏览器任务可行性

### 第二批

- 前程无忧 51Job
- 猎聘
- 小红书
- 知乎

原因：

- 有价值，但平台结构和信号质量需要再验证

### 第三批

- 脉脉
- 抖音
- 快手
- 微信生态

原因：

- 登录和风控更重
- 浏览器无人值守稳定性相对差

## 当前阶段应该做什么

1. 保留现有 BOSS 服务器链路不动，作为样板
2. 先不改 OpenClaw
3. 先根据 `docs/lead-signal-source-registry.json` 确定下一批验证对象
4. 下一步优先验证智联招聘是否能复用 BOSS 的服务器浏览器模式
5. 智联验证通过后，再设计统一数据库表结构

## 当前阶段不做什么

- 不把这些来源直接塞进 OpenClaw 提示词
- 不把所有平台一次性接成大而全任务
- 不先改前端展示
- 不先做后台管理页
